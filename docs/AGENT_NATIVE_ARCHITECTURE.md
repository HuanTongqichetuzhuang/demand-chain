# 需求链平台 — Agent 原生架构哲学

## 核心原则

**这是一个为 Agent 构建的平台，不是为人类构建的平台。**

人类通过 Agent 间接使用。Agent 是通用翻译器——把任何人类语言翻译成平台能理解的结构化数据，再把结果翻译回人类的语言。

## 语言模型

```
人类的语言（中文/英文/日文/阿拉伯语/任何语言）
    ↓ Agent 翻译
平台的语言（结构化 JSON — 唯一工作语言）
    ↓ Agent 翻译
人类收到的信息（任何语言）
```

**平台只懂一种语言：结构化数据。**

## 这意味着什么

### 对 Agent 的要求
- 你从人类那里收到什么语言，处理什么语言
- 你调用平台 MCP 工具时，全部用结构化参数（平台只认这个）
- 你收到平台的结构化返回后，翻译成人类的语言再呈现
- 28问清单不是中文也不是英文——是你在理解了平台需求模板之后，用人类最熟悉的语言去问

### 对平台的要求
- 所有 MCP 工具返回 JSON，不返回自然语言
- 所有存储的数据是结构化的（模板、分类、标注）
- 不做 UI 层面的多语言切换——不需要，Agent 做了
- web 页面只是人类可选的可视化入口，不是主要交互方式

### 对人类的好处
- 一个日文使用者，通过会日文的 Agent，无缝使用平台
- 一个阿拉伯语科学家，通过 Agent，和中方供给方 Agent 交流
- 人类永远不需要学习平台的语言——Agent 替你学了

## 结构化 — 平台的唯一语言

```
人类: "我需要一个能在800度下工作的传感器"
  ↓ Agent 翻译成平台语言
{
  "demand_type": "技术攻关",
  "industry": "传感器技术", 
  "core_need": "高温管道裂缝检测传感器",
  "technical_specs": [
    {"parameter": "工作温度", "value": 800, "unit": "°C"}
  ]
}
  ↓ 平台匹配引擎处理
{
  "match_id": "xxx",
  "score": 0.87,
  "supplier_name": "XX传感器研究所",
  "status": "pending"
}
  ↓ Agent 翻译给人类
"我帮你找到了XX传感器研究所，匹配度87%。他们在高温传感领域有12年经验。"
```

## 已做的和要改的

### 已天然符合 Agent 原生架构的
- 30+ MCP 工具全部返回 JSON
- 需求模板是结构化 JSON（不是自然语言文档）
- 分类引擎输出学科/IPC/工艺/TRL 标注
- 数据库存储 classification_json, discipline_path, ipc_codes

### 需要调整的
- ~~i18n 网页切换~~ → Agent 翻译足矣，web页面只留默认语言
- ~~28问清单是纯中文~~ → 改为 Agent 内部理解的结构化指令
- ~~README/文档的多语言~~ → 保留英文 README（开源标准），Agent_GUIDE 用结构化指令

## Agent 的 28 问 — 结构化版

原来的28问是给人类 Agent 读的中文指导。现在改为 Agent 内部指令——不管你是中文Agent还是英文Agent，都按这个结构去问：

```
Agent 内部指令（任何语言解析后等价执行）:
┌─────────────────────────────────┐
│ 任务: 确认用户身份与偏好         │
│ 字段: human_name, country, role  │
│ 方法: 用用户的母语询问           │
│ 存储: register_capability/user   │
├─────────────────────────────────┤
│ 任务: 如果是需求方              │
│ 字段: demand_raw_text            │
│ 方法: 让用户用母语自由描述      │
│ 处理: publish_demand() → JSON   │
├─────────────────────────────────┤
│ ... (其他26问同理)               │
└─────────────────────────────────┘
```

## 结论

不需要给平台做100种语言的翻译。只需要确保平台的所有输出是结构化的 JSON——然后这个地球上的任何一个 Agent 都能把它翻译成人类的语言。

**平台的语言 = JSON。不是中文。不是英文。是 JSON。**
