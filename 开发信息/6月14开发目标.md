# 2026-06-14 开发目标

> 基于 2026-06-14 项目全面审查，梳理的改进路线。
> 优先级从 P0（立即）到 P3（规划），按工程安全和核心质量排序。

## 🔴 P0 — 安全隐患（本周完成）

### 1. 密码存储升级：SHA256 → bcrypt
- **现状**：`src/web_server.py` + `src/shared/auth.py` 中 `hashlib.sha256(password.encode()).hexdigest()`，无盐
- **风险**：彩虹表可批量逆向；所有用户密码等价于明文
- **方案**：`pip install passlib[bcrypt]`，替换为 `bcrypt.hash()` / `bcrypt.verify()`
- **影响范围**：`api_register` / `api_user_password` / `login_human` / `register_human`
- **文件**：`src/web_server.py`, `src/shared/auth.py`
- **工作量**：1 小时

### 2. Token 持久化（内存 → PostgreSQL/Redis）
- **现状**：`auth.py` 中 `_tokens: dict[str, dict]` 进程内存；`task_manager.py` 中 `_tasks` 也是内存
- **风险**：服务重启 → 所有人强制下线 + 所有进行中任务丢失
- **方案**：
  - Token：建 `auth_tokens` 表（token_hash, user_id, agent_id, created_at, expires_at），配合定期清理过期行的后台任务
  - 任务：建 `async_tasks` 表（id, user_id, label, status, result, error, created_at, updated_at），替代内存 dict
- **影响范围**：`src/shared/auth.py`, `src/shared/task_manager.py`, `src/shared/models.py`
- **工作量**：4 小时

### 3. 头像存储：数据库 base64 → 文件系统
- **现状**：`api_user_avatar` 把完整 base64 data URI 写到 `User.avatar` Text 字段，一张图 ~100KB
- **问题**：头像数据挤占数据库；查询用户列表时拖慢全表；无大小/格式限制
- **方案**：
  - 上传时校验格式（jpg/png/webp）和大小（≤2MB）
  - 存到 `/app/avatars/{user_id}.jpg`
  - API 返回 URL 路径而非 data URI
  - Nginx 直接托管静态头像目录
- **影响范围**：`src/web_server.py`, `profile.html`
- **工作量**：3 小时

### 4. 注册限速 + 邮箱验证
- **现状**：`/api/register` 无验证码、无 IP 限流、无邮箱确认
- **风险**：可被批量注册刷库，消耗 API 配额
- **方案**：
  - FastAPI 中间件 `slowapi` 限流（同一 IP 每小时 ≤5 次注册）
  - 注册后发送含验证链接的邮件，邮箱验证通过后才可登录
  - 未验证邮箱≥7 天自动清理
- **影响范围**：`src/web_server.py`, `pyproject.toml`
- **工作量**：3 小时

### 5. 用户输入安全加固
- **现状**：`api_user_stats` 中 `ForumTopic.agent_id.ilike('%'+email+'%')` 拼接用户输入（虽 ORM 层有转义，但 pattern 参数未净化）
- **方案**：改用参数化绑定 + 对 `email` 做格式校验（正则 `^[^@]+@[^@]+\.[^@]+$`）
- **影响范围**：`src/web_server.py` 中所有查询参数处理
- **工作量**：1 小时

## 🟠 P1 — 匹配引擎质量（1-2 周）

### 6. 匹配引擎 limit(500) 硬编码修复
- **现状**：`src/matching/engine.py:24` — `select(CapabilityProfile).limit(500)`
- **问题**：供应商已达 1788 家，此限制会漏掉 70%+ 的候选
- **方案**：移除硬编码 limit，改为通过 `run_matching()` 参数控制（默认全量），配合分页处理
- **影响范围**：`src/matching/engine.py`, `src/worker.py`
- **工作量**：0.5 小时

### 7. pgvector 语义搜索接入
- **现状**：
  - 数据库是 `pgvector/pgvector:pg16`，支持向量搜索
  - `Demand.embedding` 和 `CapabilityProfile.embedding` 字段已建好（Vector(1536)）
  - 但无任何代码写入或查询 embedding，全部靠 TF-IDF
- **改进**：
  - 新增 `generate_embedding(text)` → 调 DeepSeek embedding API 返回 1536 维向量
  - 需求入库 / 供应商入库时自动生成并存储向量
  - 匹配引擎新增语义通道：先 TF-IDF 粗筛（召回），再 pgvector 余弦距离精排（排序）
  - 语义搜索在中文场景下准确率显著优于 TF-IDF（"高温传感器"↔"耐热测量设备"也能匹配）
- **影响范围**：`src/adapters/llm_client.py`（新增 embedding 端点）、`src/matching/engine.py`、`src/demand/service.py`、`src/shared/semantic_search.py`
- **工作量**：8 小时

### 8. Worker 健康检查 + 失败告警
- **现状**：`src/worker.py` 每 15 分钟轮询，匹配失败仅 `traceback.print_exc()`
- **方案**：
  - 暴露 `/health` HTTP 端点，Docker 健康检查定期探测
  - 连续匹配失败 ≥3 次 → 发送告警通知（Server酱/企业微信）
  - 记录每次匹配耗时到日志，超时（>60s）单独告警
- **影响范围**：`src/worker.py`, `docker-compose.yml`
- **工作量**：2 小时

### 9. docker-compose 补齐 Web 服务
- **现状**：`docker-compose.yml` 只有 db + mcp + worker，web_server 独立运行
- **方案**：添加 web 容器，与 mcp 共享代码卷，暴露 8080 端口
- **影响范围**：`docker-compose.yml`, `Dockerfile`
- **工作量**：1 小时

## 🟡 P2 — 架构与性能（1-2 周）

### 10. Redis 缓存接入
- **现状**：每次请求直查 PostgreSQL，首页统计数、分类列表等热门数据反复计算
- **方案**：
  - docker-compose 加 Redis 容器
  - 缓存首页统计（TTL 5 分钟），匹配结果缓存 30 分钟
  - 缓存失效策略：数据变更时主动清除相关缓存 key
- **影响范围**：`docker-compose.yml`, `src/shared/cache.py`（新建）, `src/web_server.py`
- **工作量**：6 小时

### 11. 前端错误处理完善
- **现状**：多处 `catch(e){}` 静默吞掉错误（`profile.html` 的 `loadUserData()`, `loadUserStats()` 等）
- **方案**：
  - 统一错误 toast 组件（已存在 `.msg-err` 样式）
  - 网络错误 / API 错误在前端显示"加载失败"提示
  - global error handler 捕获未处理的 Promise rejection
- **影响范围**：所有 HTML 页面 + `shared.css`
- **工作量**：2 小时

### 12. Nav 缓存问题根治
- **现状**：`nav.js` 被浏览器缓存，改导航后用户需 `Ctrl+Shift+R` 硬刷新
- **方案**：HTML 中引用时加自动版本号 `<script src="nav.js?v={BUILD_TIMESTAMP}">`
- **影响范围**：所有 HTML 页面
- **工作量**：1 小时

### 13. 测试覆盖率提升
- **现状**：`tests/` 只有基础 E2E 测试，无匹配/分类/爬虫的独立测试
- **新增**：
  - `test_matching.py` — 匹配引擎单测（mock DB session）
  - `test_classification.py` — 分类引擎单测（mock LLM 返回）
  - `test_crawler.py` — 爬虫解析单测（mock HTTP 响应）
  - `test_auth.py` — 注册/登录/token 验证测试
- **影响范围**：`tests/` 目录
- **工作量**：8 小时

### 14. Alembic 迁移初始化
- **现状**：`alembic.ini` + `alembic/env.py` 存在但无迁移记录
- **方案**：生成初始迁移，后续改模型强制走 `alembic revision --autogenerate`
- **影响范围**：`alembic/versions/` 目录
- **工作量**：1 小时

## 🟢 P3 — 战略功能（规划阶段）

### 15. 需求链分解引擎
- **概念**：从"需求←匹配→供给"二元关系升级为真正的**链式分解**
- **示例**：
  ```
  大需求："造火箭"
    → 子需求1："液体燃料发动机"       → 匹配发动机供应商
    → 子需求2："星载导航控制系统"      → 匹配电子/软件供应商
    → 子需求3："耐高温外壳材料"       → 匹配材料供应商
    → 子需求4："地面测试设备"         → 匹配测试设备供应商
  ```
- **现状**：`Demand.parent_id` 字段已存在，但没有分解逻辑和 UI
- **工作量**：16 小时（分解算法 + 可视化展示）

### 16. A2A Agent 通信原型
- **现状**：Agent 协议文档 11 章完善（五级信息分类 L0-L4、端到端加密、30 天过期），但代码层无实现
- **Phase 2 起点**：
  - 实现 A2A "握手"原型：Agent A → 平台 → Agent B，交换 L0 公开信息
  - 匹配成功后自动建立 A2A 会话通道
  - 共享 Agent Card（`/.well-known/agent.json`）
- **工作量**：20 小时

### 17. 数据飞轮
- **现状**：数据单向流动（爬虫→DB→匹配→结束），结果不反哺算法
- **方案**：
  - 匹配后跟踪"是否联系→是否合作→是否成功"
  - 成功的匹配提高对应分类/标签的权重
  - 失败的匹配降低对应分类的权重
- **工作量**：12 小时
